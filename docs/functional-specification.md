## 1. История проекта

Проект «Редактор блок-схем» был инициирован в рамках учебного курса «Проектирование и архитектура программных систем». Его целью является создание инструмента, позволяющего автоматизировать процесс преобразования исходного кода в блок-схемы, а также обеспечивать возможность их последующего редактирования и рецензирования. Основной задачей разработки стало снижение трудозатрат при подготовке технической документации и отчётов по программным проектам.

На начальном этапе было проведено исследование существующих инструментов генерации диаграмм и их интеграции с исходным кодом. Было выявлено, что большинство решений не обеспечивают обратную синхронизацию между кодом и схемой. Это определило необходимость создания собственного веб-приложения, обеспечивающего более тесную связь между исходным кодом и визуальной моделью алгоритма.

В процессе проектирования было принято решение использовать трёхслойную архитектуру, что обеспечило разделение логики приложения на серверную часть, отвечающую за обработку кода и взаимодействие с сервером базы данных, и клиентскую часть, реализующую визуализацию и работу пользователя с блок-схемами. Серверная часть создаётся на основе ASP.NET Core Web API, клиентская — с использованием HTML, CSS и JavaScript. В качестве системы управления данными выбрана база PostgreSQL.

На этапе планирования также были определены ключевые функциональные модули:
- Модуль парсинга кода, обеспечивающий анализ исходного текста программы и построение промежуточного представления;
- Модуль генерации блок-схем, формирующий визуальные элементы на основе синтаксического дерева;
- Модуль рецензирования, предназначенный для добавления комментариев и предложений к блок-схеме;
- Модуль синхронизации, обеспечивающий обновление кода при внесении изменений в диаграмму.

Разработка проекта выполняется командой из шести участников, включающей backend- и frontend-разработчиков, тестировщика и аналитика-документатора. Общая продолжительность проекта составляет пять месяцев, по завершении которых будет представлена демонстрационная версия приложения и итоговая документация.  
Проект прошёл стадии анализа, архитектурного проектирования и документирования.

### 2.1. Требования пользователя

Раздел «Требование пользователя» перечисляет выявленные требования к решению с точки зрения заказчика и конечных пользователей.

В результате анализа предметной области и взаимодействия с заказчиком были определены основные категории пользователей системы и их ключевые потребности.  
Система «Редактор блок-схем» ориентирована на специалистов, работающих с программным кодом и технической документацией, и должна обеспечивать простоту работы, визуальную наглядность и возможность совместного рецензирования результатов.

Основными пользователями системы являются проект-менеджер, разработчик и рецензент. Каждая категория пользователей имеет собственный набор требований к функциональности и интерфейсу системы.

Требования проект-менеджера:
1. Возможность просматривать текущие версии блок-схем и сопоставленный им исходный код.
2. Доступ к журналу изменений и комментариев, оставленных рецензентами и разработчиками.
3. Возможность утверждения или отклонения предложенных изменений и замечаний.
4. Просмотр состояния проекта (готовность, этап выполнения, версия релиза).
5. Удобный интерфейс для визуального контроля структуры блок-схем и истории версий.

Требования разработчика:
1. Автоматическая генерация блок-схем на основе исходного кода C++.
2. Возможность ручного редактирования элементов диаграммы (добавление, удаление, изменение связей).
3. Синхронизация изменений между кодом и визуальной моделью алгоритма.
4. Отображение специальных комментариев из исходного кода как границ или аннотаций блоков.
5. Поддержка загрузки и сохранения исходного кода и схем через интерфейс системы.
6. Просмотр и принятие/отклонение комментариев рецензентов с автоматическим обновлением схемы.
7. Удобное визуальное представление диаграмм (масштабирование, панорамирование, группировка элементов).

Требования рецензента:
1. Возможность добавлять комментарии к элементам блок-схемы или строкам кода.
2. Возможность предлагать исправления в визуальной схеме с привязкой к конкретному элементу.
3. Просмотр истории правок, оставленных другими пользователями.
4. Получение уведомлений об изменениях в схеме или коде после рецензирования.
5. Возможность фильтрации комментариев по статусу («новый», «в работе», «принят», «отклонён»).

Общие требования к интерфейсу:
1. Интерфейс должен быть интуитивно понятным, визуально чистым и не перегруженным элементами.
2. Все действия (добавление комментариев, редактирование, подтверждение изменений) должны выполняться через графические элементы управления, без необходимости ручного ввода команд.
3. Система должна обеспечивать визуальную обратную связь на все действия пользователя (например, подсветка изменённых элементов, уведомления об ошибках).
4. Интерфейс должен поддерживать масштабирование и адаптивное отображение схем в окне браузера.
5. Для повышения удобства предусмотрены всплывающие подсказки, панели инструментов и контекстное меню.

### 2.2. Системные требования

Аппаратные требования для клиентской части:
- Процессор: не ниже Intel Core i3 / AMD Ryzen 3.
- Оперативная память: не менее 4 ГБ.
- Подключение к сети Интернет со скоростью не ниже 1 Мбит/с.

Аппаратные требования для серверной части:
- Процессор: Intel Core i5 / AMD Ryzen 5 или выше.
- Оперативная память: не менее 8 ГБ.
- Свободное дисковое пространство: не менее 100 ГБ.
- Стабильное интернет-соединение и возможность развёртывания ASP.NET Core Web API.

Программные требования (клиентская часть):
- Операционные системы: Windows 10/11, Linux, macOS.
- Браузеры: Google Chrome, Microsoft Edge, Mozilla Firefox.
- Технологии: HTML5, CSS3, JavaScript.

Программные требования (серверная часть):
- Среда выполнения: .NET 8.0 / ASP.NET Core Web API.
- СУБД: PostgreSQL (через драйвер Npgsql).
- Среда разработки: Visual Studio / Visual Studio Code.
- Система контроля версий: GitHub.
- Инструменты тестирования: Postman, Swagger, GitHub Actions.

## 4. Предположения и зависимости

Раздел фиксирует ключевые предположения, сделанные при разработке функциональной спецификации, а также внешние и внутренние зависимости, влияющие на реализацию и внедрение проекта. Указанные положения помогают определить рамки ответственности команды и снизить риски, связанные с изменением исходных условий выполнения проекта.

Предположения:
1. Доступность инфраструктуры. Предполагается, что во время разработки и тестирования участники команды будут иметь постоянный доступ к интернету, средствам совместной работы и средам разработки.
2. Стабильность состава команды. Состав проектной группы остаётся неизменным в течение всего периода реализации проекта. Каждая роль выполняется одним закреплённым участником, без необходимости замены или перераспределения обязанностей.
3. Ограниченный учебный контекст. Проект разрабатывается в учебных целях и не предназначен для промышленной эксплуатации. Это допускает упрощённые требования к безопасности, отказоустойчивости и масштабируемости системы.
4. Доступность технологий. Все используемые программные средства, библиотеки и фреймворки являются открытыми или предоставляются бесплатно в рамках образовательных лицензий.
5. Определённость предметной области. На момент проектирования функциональность, поддерживаемые языки программирования и структура блок-схем согласованы с заказчиком и не подлежат значительным изменениям в ходе текущего релиза.
6. Единые стандарты разработки. Все участники проекта следуют единым правилам оформления кода, ведения документации и регистрации изменений в системе контроля версий. Это обеспечивает согласованность и воспроизводимость результатов.

Зависимости:
1. Технические зависимости. Работа системы зависит от стабильности внешних библиотек и компонентов. Нарушение совместимости версий или обновление платформы может потребовать корректировок в исходном коде.
2. Организационные зависимости. Сроки выполнения проекта напрямую зависят от учебного расписания, промежуточных проверок и графика защиты. Сдвиг учебных этапов или задержка согласования требований может повлиять на общий календарный план.
3. Зависимость от инфраструктуры хранения и совместной работы. Система контроля версий и облачное хранилище являются критически важными элементами рабочей среды. Недоступность этих сервисов может временно приостановить процесс разработки или тестирования.
4. Зависимость от согласований заказчика. Реализация отдельных функций (например, модификации интерфейса, изменения логики синхронизации) выполняется только после подтверждения заказчиком. Задержка обратной связи способна повлиять на сроки реализации.

## 5. Проект решения

### 5.1. Концептуальный проект

Концептуальный проект определяет общие принципы организации системы и логику взаимодействия её основных компонентов. Документ отражает архитектуру будущего решения в терминах, понятных заказчику и пользователям, без использования избыточной технической терминологии. Цель данного раздела — продемонстрировать, каким образом система будет реализовывать ключевые сценарии, описанные в разделе «Сценарии использования».

Общая концепция решения представляет собой веб-приложение клиент-серверного типа, обеспечивающее возможность автоматической генерации, редактирования и рецензирования блок-схем, связанных с исходным кодом программ на языке C++ и Pascal. Пользователь взаимодействует с системой через веб-интерфейс, отправляя запросы к серверу, где происходит обработка данных, синхронизация с исходным кодом и хранение результатов.

Система разделена на три логических слоя, взаимодействующих между собой через чётко определённые интерфейсы:
1. Клиентская часть:  
   Отвечает за отображение и интерактивное взаимодействие с пользователем. Через веб-интерфейс пользователь может:  
   - просматривать, редактировать и комментировать блок-схемы;  
   - загружать и скачивать диаграммы;  
   - получать уведомления о новых рецензиях или изменениях;  
   - управлять доступом и ролями в проекте (для проект-менеджеров).
2. Серверная часть:  
   Сервер обрабатывает запросы клиентов, выполняет анализ исходного кода, управляет версионностью и синхронизацией, а также обеспечивает взаимодействие между модулями системы (парсер, генератор, рецензирование).
3. Хранилище данных:  
   В базе данных хранятся:  
   - исходные файлы кода;  
   - структура и элементы блок-схем;  
   - комментарии и рецензии;  
   - пользовательские данные (аккаунты, роли, статусы задач);  
   - история изменений и версии диаграмм.

### 5.2. Логический проект

Основные подсистемы:
1. Подсистема управления пользователями  
   Обеспечивает регистрацию, аутентификацию, управление ролями и доступом к функциональности системы.  
   Ключевые объекты:  
   - `User` — пользователь системы.  
   - `Role` — роль пользователя (администратор, проект-менеджер, разработчик, рецензент).  
   - `AccessManager` — контролирует права доступа к проектам и действиям.
2. Подсистема парсинга и анализа кода  
   Отвечает за обработку исходного кодa C++, выделение структурных элементов и создание внутреннего представления программы.  
   Ключевые объекты:  
   - `CodeFile` — исходный файл программы.  
   - `Parser` — модуль синтаксического анализа.  
   - `ASTNode` — элемент абстрактного синтаксического дерева.  
   - `CommentMarker` — объект, определяющий границы блок-схем с помощью специальных комментариев.
3. Подсистема генерации и редактирования блок-схем  
   Формирует визуальные блок-схемы на основе данных парсера и обеспечивает возможность их редактирования.  
   Ключевые объекты:  
   - `Flowchart` — объект блок-схемы.  
   - `Block` — элемент схемы (оператор, условие, цикл).  
   - `Connector` — связь между блоками.  
   - `FlowchartEditor` — интерфейс редактирования блок-схем.
4. Подсистема рецензирования  
   Обеспечивает возможность добавления и обработки замечаний и предложений по блок-схемам или исходному коду.  
   Ключевые объекты:  
   - `Review` — рецензия или комментарий.  
   - `ReviewStatus` — статус рецензии (в работе, принято, отклонено).  
   - `ReviewManager` — обработка и согласование рецензий.
5. Подсистема синхронизации  
   Реализует двустороннюю связь между кодом и диаграммами, обеспечивает актуальность представлений.  
   Ключевые объекты:  
   - `SyncManager` — координация изменений между схемой и кодом.  
   - `VersionControl` — хранение версий файлов и диаграмм.  
   - `ChangeTracker` — отслеживание изменений и фиксация версий.
6. Подсистема хранения данных  
   Определяет модели для хранения информации о пользователях, рецензиях, коде и диаграммах.  
   Ключевые объекты:  
   - `Database` — логическое представление БД.  
   - `Project` — совокупность исходных файлов, диаграмм и рецензий, объединённых в единый контекст.  
   - `FileStorage` — компонент, отвечающий за физическое хранение файлов.

Описание взаимодействия:
- Пользователь (разработчик или рецензент) вносит изменения в код или диаграмму.  
- `SyncManager` фиксирует изменение и вызывает соответствующие методы `Parser` и `Flowchart` для обновления данных.  
- Все версии сохраняются через `VersionControl`, обеспечивая откат изменений.  
- `ReviewManager` отслеживает рецензии и уведомляет участников о статусе через систему уведомлений.  
- `Project` выступает связующим элементом между всеми сущностями, храня общие данные.

### 5.3. Физический проект

Физический проект представляет собой проекцию логического проекта на имеющееся аппаратное, программное и технологическое окружение. Так, нам надлежит выбрать те или иные технологии для реализации идей, заложенных в концептуальном и уточненных в логическом дизайне.

Физический проект фактически представляет собой документацию с четким указанием параметров реализуемой функциональности в терминах разработчиков программного обеспечения.

Кроме того, решаются как вопросы создания компонентов, допускающих повторное использование, так и вопросы применения написанного ранее кода (или сторонних разработок).  
Ниже приведён физический проект решения «Редактор блок‑схем», соответствующий принятой трёхслойной архитектуре и ограниченному учебному контексту.

#### 5.3.1. Технологический стек
- Серверная часть (`./src/WebApi`, ./src/Application`, `./src/Domain`, `./src/Infrastructure`):
  - Платформа: .NET 8 (ASP.NET Core Web API).
  - Язык: C# 12.
  - ORM и доступ к данным: Entity Framework Core + `Npgsql` (PostgreSQL).
  - Миграции БД: EF Core Migrations.
  - Валидация: FluentValidation (в составе слоя `Application`).
  - Логирование: Serilog (консоль + файл), уровни логирования конфигурируемы.
  - Документация API: Swagger (Swashbuckle) с описанием DTO, схем и примеров.
  - Тестирование: xUnit/MSTest (юнит‑тесты для `Domain`/`Application`), Postman/REST Client — интеграционные снэпшоты.
- Клиентская часть:
  - Веб‑интерфейс: HTML5, CSS3, JavaScript.
  - Визуализация блок‑схем: Mermaid.js (рендеринг диаграмм, экспорт в SVG/PNG).
  - Сборка: ~TODO~.
- База данных:
  - PostgreSQL 15+.
  - Схема `public`, именование таблиц в CamelCase.

#### 5.3.2. Соответствие логических слоёв физической структуре
Структура исходников и сборок следует шаблону «чистой архитектуры»:
- ./src/Domain` → сборка `DAOSS.Domain.dll`  
  Сущности домена, value‑объекты, интерфейсы репозиториев, доменные события.
- ./src/Application` → `DAOSS.Application.dll`  
  Use‑cases (команды/запросы), обработчики, интерфейсы сервисов, маппинги DTO.
- ./src/Infrastructure` → `DAOSS.Infrastructure.dll`  
  Реализации репозиториев EF Core, конфигурации DbContext/моделей, интеграции.
- ./src/WebApi` → `DAOSS.WebApi.dll`  
  Транспортный слой (контроллеры, фильтры, DI‑композиция, Swagger, аутентификация).

Сборка и публикация:
- Конфигурации: `Debug` (локальная разработка), `Release` (демо/стенд).

#### 5.3.3. Физическая модель данных (уровень БД)
- Таблицы (пример целевого набора):
  - `users` (id, email, password_hash, display_name, role, created_at, updated_at).
  - `projects` (id, name, description, owner_user_id, created_at, updated_at).
  - `code_files` (id, project_id, filename, language, content, version, created_at, updated_at).
  - `flowcharts` (id, project_id, name, layout_json, version, created_at, updated_at).
  - `blocks` (id, flowchart_id, type, text, position_json, meta_json).
  - `connectors` (id, flowchart_id, from_block_id, to_block_id, label, meta_json).
  - `reviews` (id, project_id, author_user_id, target_type, target_id, status, comment, created_at, resolved_at).
  - `audit_logs` (id, project_id, actor_user_id, action, payload_json, created_at).
- Связи и индексы:
  - FK: `projects.owner_user_id → users.id`, `code_files.project_id → projects.id`, `flowcharts.project_id → projects.id`, `blocks.flowchart_id → flowcharts.id`, `connectors.flowchart_id → flowcharts.id`, `reviews.project_id → projects.id`.
  - Индексы: `idx_code_files_project_id`, `idx_flowcharts_project_id`, `idx_reviews_target`, `idx_audit_project_time`.
- Хранение контента:
  - Исходный код хранится в `code_files.content` (TEXT). Для больших файлов возможно вынесение в файловое хранилище (`FileStorage`) с ссылкой.
  - Геометрия/оформление блок‑схем — в `layout_json`, `position_json`, `meta_json` (JSONB).
- Миграции:
  - EF Core Migrations под SCM (`dotnet ef migrations add <name>` → `dotnet ef database update`).
  - Версионирование схемы — через инкрементальные миграции, автоматический прогон в CI на тестовой БД.

#### 5.3.4. Контракты API (срез)
- Базовый URL: `/api/v1`.
- Аутентификация: JWT Bearer (`Authorization: Bearer <token>`).
- Примеры конечных точек:
  - `POST /auth/register`, `POST /auth/login`.
  - `GET /projects`, `POST /projects`, `GET /projects/{id}`, `DELETE /projects/{id}`.
  - `GET /projects/{id}/files`, `POST /projects/{id}/files` (multipart), `GET /files/{id}`, `PUT /files/{id}`.
  - `POST /flowcharts/generate` (из кода), `GET /projects/{id}/flowcharts`, `PUT /flowcharts/{id}`.
  - `POST /reviews`, `GET /projects/{id}/reviews`, `PATCH /reviews/{id}`.
  - `GET /export/flowcharts/{id}?format=svg|png`.
- Документация: `/swagger` (JSON и UI), примеры запросов/ответов, схемы DTO, коды ошибок.

#### 5.3.5. Безопасность и управление доступом
- Модель ролей: `Admin`, `ProjectManager`, `Developer`, `Reviewer` (см. `Domain/Role`, `Application` правила авторизации).
- Аутентификация: JWT (симметричный ключ в `appsettings.{Environment}.json`/секретах окружения).
- Авторизация: политики/атрибуты на контроллерах и хендлерах (например, `[Authorize(Roles = "ProjectManager,Admin")]`).
- Хранение паролей: `password_hash` с использованием `PBKDF2`/`Argon2` (через библиотеку, без самописной криптографии).
- Лимитирование и защита: базовое rate‑limit (ASP.NET Core Rate Limiting), валидация входных данных, размер загрузок, CORS по списку источников.

#### 5.3.6. Логирование, мониторинг, аудит
- Серверные логи: Serilog → консоль и файл (`logs/app-.log`), ротация по дате/размеру.
- Корреляция запросов: `X-Correlation-ID` + middleware.
- Аудит доменных событий: записи в `audit_logs` с указанием пользователя, времени, типа действия и payload.\
